# Tailwind CSS

![Theo's thought on CSS](/images/theo-css.png 'What Theo thinks about CSS')

CSS - это боль. На этом можно было бы закончить статью, но как бы мы того ни
хотели, нам приходится иметь дело с каскадными таблицами стилей. Кто-то сдается
и тянет на проект **Bootstrap**, кто-то пытается писать семантический CSS,
соблюдая **BEM** или похожую методологию, кто-то вообще берет готовые UI
компоненты, стараясь забыть про CSS, как про страшный сон.

Сегодня я хочу рассказать про то, как за 3 года написания стилей сначала с
использованием чистого CSS, затем с помощью препроцессора **SCSS**, стараясь
соблюдать методологию BEM, а потом применяя **CSS Modules**, я в конце концов
пришел к **Tailwind CSS**

Но начнем издалека.

## Полезный опыт и показательный пример

Год назад я участвовал в конкурсе, в рамках которого нужно было разработать
небольшой виджет с использованием библиотеки **React**. Суть виджета заключалась
в том, что пользователю дается возможность пройти опрос, в результате которого
приложение построит API запрос на получение товара, подходящего клиенту.

[Ссылка на демо](https://wardxela.github.io/optimax-widget-example/)

Виджет представляет из себя приложение, состоящее из 10 очень **похожих**, но
все же **разных** экранов, требующих от пользователя какой-то выбор.

В качестве инструмента для написания стилей я выбрал препроцессор **SCSS**, а
также воспользовался инкапсуляцией стилей с помощью **CSS Modules**.

По началу все шло очень даже неплохо, я старался переиспользовать готовые стили
как мог, выделял похожие паттерны и объединял их семантически, но чем ближе
проект двигался к завершению, стили превращались во все больший ад.

Основная проблема, с которой я столкнулся - **очень** большое разнообразие
отступов между разными компонентами. В одном случае между текстом отступ в
`30px`, в другом - `20px`, а в третьем - `15px`.

Какие у меня были варианты решения этой проблемы?

- Создать кучу модификаторов для классов, сохраняя семантическое значение стилей

  ```SCSS
  .card {
    &__text {
      /* ... */
      &_mb-large {
        margin-bottom: 30px;
      }
      &_mb-medium {
        margin-bottom: 20px;
      }
      &_mb-small {
        margin-bottom: 15px;
      }
    }
  }
  ```

  Мало того, что эти стили на самом деле не несут никакого смысла (об этом еще
  пойдет речь дальше), так еще и если мне понадобится поставить отступ в `10px`,
  мне придется создавать класс вроде `&_mb-extra-small`. А что на счет
  переиспользуемости? Что, если мне захочется у компонента `post` задать
  аналогичные стили? Мало того, что мне придется дублировать код, так еще и у
  стилей не будет присутствовать какая-либо система (об этом тоже позже).

- Создать общий класс-утилиту, который будет содержать все возможные значения
  для отступов

  ```scss
  .OWMargin {
    &Top {
      &-Auto {
        margin-top: auto;
      }

      @for $i from 0 through 4 {
        $value: 25 + 5 * $i;
        &-#{$value} {
          margin-top: $value + px;
        }
      }

      &-100 {
        margin-top: 100px;
      }
    }

    &Bottom {
      @for $i from 0 through 8 {
        $value: 10 + 5 * $i;
        &-#{$value} {
          margin-bottom: $value + px;
        }
      }
    }
  }
  ```

  Это решение уже лучше, поскольку класс можно переиспользовать многократно в
  разных местах в проекте. Остается лишь только проблема системности стилей, так
  как допуская слишком большой разброс возможных значений, теряется контроль над
  общей системой дизайна в приложении.

  Но я привел пример лишь с отступами. А что на счет других свойств? Создавать
  утилиты под каждое свойство? Поздравляю, вы придумали Tailwind CSS!

## Знакомство с Tailwind

На самом деле я, как и многие, не воспринимал **Tailwind** всерьез. Особенно
после печального опыта с **Bootstrap**, подобные решения не кажутся хоть сколько
нибудь стоящими. Но Tailwind смог поразить меня в первую очередь своим взглядом
на CSS. Если сейчас меня попросят сверстать проект на чистом CSS, я начну
изобретать свой собственный Tailwind.

Окончательной причиной моего знакомства с этим фреймворком стало прочтение
статьи [Adam Wathan'a](https://adamwathan.me/) - автора самой библиотеки
Tailwind CSS. Для тех, кому стало интересно,
[вот ссылка на саму статью](https://adamwathan.me/css-utility-classes-and-separation-of-concerns/).
По сути все мысли, которые я изложу в своей статье высказаны и там.

## CSS не имеет ничего общего с семантикой

Главная проблема большинства подходов к CSS заключается в том, что люди пытаются
придать стилям смысловое значение. Делается это в попытке создания уникальных
стилей для каждого компонента. Таким образом по началу удается поддерживать
изоляцию стилей друг от друга и относительно просто вносить в них изменения.

Но проблемы начинают появляться как только мы имеем дело с приложением,
состоящим более чем из 1-2 страниц.

Простой пример: на сайте есть компонент с карточкой товара. Мы создаем для него
класс `.card` и описываем все стили, отталкиваясь от него. Но что, если на сайте
появляется другая карточка, очень похожая на текущую, но все же имеющую ряд
отличий. Здесь могут помочь **BEM** модификаторы, но если вариаций становится
слишком много (как в примере с виджетом), то код начнет превращаться во что-то
подобное:

```css
.card {
  /* ... */
}
.card-2 {
  /* ... */
}
.card-fancy {
  /* ... */
}
```

Создатели Tailwind CSS предлагают отказаться от семантического CSS, так как
очень часто название класса совершенно не передает никакого смысла.

Думаю всем знакома такая картина в HTML

```HTML
<div class="card">
  <div class="card__outer-wrapper">
    <div class="card__inner-wrapper">
      <div class="card__container">
        И дальше до бесконечности
      </div>
    </div>
  </div>
</div>
```

Мало того, что подобных классов в проекте встречается очень много, так они еще и
совершенно не поддаются переиспользованию и зачастую просто дублируют друг
друга.

Не проще ли в своем CSS один раз написать это:

```css
.flex {
  display: flex;
}
```

Чем создавать сотню одинаковых классов, вроде этих:

```css
.card__wrapper {
  display: flex;
}
.post__wrapper {
  display: flex;
}
.comment__wrapper {
  display: flex;
}
```

## Utility first approach. Утилиты как основной способ построения стилей

Взамен семантическому CSS создатели Tailwind предлагают делать классы абсолютно
абстрактными и совершенно не описывающими то, что подразумевается под тем или
иным куском html кода.

Пример

```html
<div class="grid grid-cols-2 gap-6">
  <div class="flex flex-col items-center gap-2">
    <!-- ... -->
  </div>
  <div class="flex flex-col items-center gap-2">
    <!-- ... -->
  </div>
</div>
```

Из представленного выше примера невозможно догадаться о том, что подразумевает
автор. Может быть это секция с товарами, а может быть разметка для секции с
приветствием.

Интересно то, что зачастую нам и не надо передавать смысл через стили. Если мы
хотим придать какой то части кода на html семантическое значение, мы можем
добиться этого с помощью шаблонов или компонентов, о которых пойдет речь дальше,
но в большинстве же случаев, нам даже не придется этого делать.

Благодаря тому, что мы начинаем стилизовать свой сайт с помощью набора готовых
утилит, мы полностью избавляемся от файлов CSS. Нам больше не нужно изобретать
тысячу бессмысленных классов, решающих +- похожие задачи. Теперь, чтобы изменить
какое-то свойство у отдельного, нам не нужно искать нужный класс среди кучи
других, мы просто добавляем класс-утилиту и оно работает.

Но помимо вышеперечисленных преимуществ, Tailwind дает самое главное - он
ограничивает (в хорошем смысле) разработчика в возможной палитре значений для
каждого свойства. Таким образом, стили становятся консистентными, структурными,
не конфликтующими и образующими единую систему, на основе которой становится
просто невозможно сделать что-то некрасиво или неправильно.

## Система в дизайне

Я написал уже несколько проектов на Tailwind, и могу с уверенностью сказать, что
если мне нужно будет изменить что-то в дизайне, я с легкостью справлюсь с этой
задачей. Я запросто смогу изменить в нужном месте цвет фона, где-то поставить
удобный отступ, а где-то и во все изменить разметку скажем с флексов на гриды,
не повредив при этом остальную часть сайта. Сможет ли тем же похвастаться
проект, написанный, скажем, на том же SCSS? Я бы хотел на это посмотреть.
